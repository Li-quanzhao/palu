# 帕鲁 — AI 售后小助手

广州睿程信息科技有限公司 · 内部 AI 售后客服机器人

**核心理念**：能检索的不生成，有缓存的不调用，能规则判断的不上 Agent，无关问题不花钱。

---

## 一、项目背景

### 业务痛点

公司售后群里每天有人问重复问题——"软件闪退怎么办"、"License 怎么激活"、"报错 XYZ 什么意思"。人工翻文档回复太慢，客户等久了容易抱怨。

### 目标

做一个群里的 AI 帕鲁：@它一下，它自己翻知识库 + 查工单状态来回答。

### 售后场景说明

睿程科技是 B2B 项目交付公司（给别的企业做软件）。售后对象是**客户公司的 IT 人员或系统管理员**，不是普通消费者。所以帕鲁知识库里的内容是：项目常见报错、License 激活、功能使用问题、环境配置问题。不是退货退款、商品咨询这类 C 端售后。

### 命名由来

"帕鲁"取自《幻兽帕鲁》（Palworld）—— 帮你干活的小助手。内部项目代号，非正式产品名。

## 二、实现路径（5 天迭代）

| Day | 版本 | 能力 | 核心代码 |
|:---:|:----|:-----|:---------|
| 1 | 终端问答 | API 调用、System Prompt、Temperature 调参 | `palu.py` |
| 2 | RAG 知识库 | Embedding 向量检索、余弦相似度、保底阈值 | `palu_rag.py` + `售后知识库.json` |
| 3 | Agent 实验 | ReAct 循环、Tool Calling、注入防御、输出脱敏 | `palu_agent.py` |
| 4 | 钉钉机器人 | 钉钉回调模块、Flask Blueprint 集成、签名验证 | `dingtalk_bot.py` + `app.py` |
| 5 | 生产化 | 热更新守护进程、后台管理脚本、API 认证、限流、缓存、看板、反馈飞轮 | `app.py` + `palu_watcher.py` |

### 架构决策记录

| 决策点 | 选择 | 理由 |
|:-------|:-----|:------|
| 工单查询 | 正则路由（非 Agent） | 规则明确、毫秒级、零成本。不硬上 Agent |
| 工作范围检测 | 关键词拦截 + 兜底 LLM | 零成本前置拦截，敏感词命中直接拒绝 |
| 认证方式 | X-API-Key 静态 Token | 内部工具、调用方单一，不用 OAuth/JWT |
| 管理看板 | Basic Auth | 浏览器原生支持，无需前端写登录逻辑 |
| 生产服务器 | 腾讯云 + waitress | Flask dev server 不适合生产，腾讯云 + waitress 多线程稳定 |

## 三、功能总览

### 🎯 核心问答

- **RAG 知识库检索**：36 条 B2B FAQ，8 个类别，覆盖部署/License/报错/运维/售后流程
- **三层保底机制**：
  - 相似度 ≥ 0.85 → 直接返回原文（零 Token 消耗）
  - 0.65 ≤ 相似度 < 0.85 → LLM 基于检索结果生成回答
  - 相似度 < 0.65 → 礼貌表示"不知道"，记录到未答问题日志
- **工单查询**：正则匹配"工单+数字"，查模拟工单数据（支持 1024/1025/1026）
- **多轮对话**：自动保留上下文（最多 10 轮），指代"它""那个"能正确理解
- **连续低分自动转人工**：同 session 内连续 3 次低分 → 建议联系人工客服

### 🔒 安全防护

| 防护层 | 机制 | 效果 |
|:-------|:-----|:------|
| 输入长度限制 | 500 字符截断 | 防超长输入攻击 |
| 敏感信息过滤 | 入口处正则替换 API Key/邮箱/手机号 | 不进日志、不进 LLM |
| Prompt 注入检测 | 16 条中英文正则规则（忽略命令/忽略指令/越狱等） | 拦截后返回固定提示 |
| 禁答主题检测 | 薪资/密码/源代码/其他客户信息/政治/宗教 6 类 | 匹配则直接拒绝 |
| 工作范围检测 | 无关关键词（天气/新闻/股票/游戏/明星/八卦等）拦截 + LLM 兜底 | 防止被刷无关问题 |
| 输出脱敏 | 正则替换 API Key/邮箱/手机号/身份证 | 不拦截只替换，不误伤正常回复 |
| API 降级 | DeepSeek 连续 3 次失败 → 切 Qwen-turbo | 保证服务可用性 |
| Embedding 降级 | 向量检索失败 → 关键词搜索降级 | 知识库可用性零中断 |

### 🚦 流量治理

| 机制 | 限制 | 说明 |
|:-----|:------|:------|
| 全局限流 | 20 次/分钟/IP | 总请求频率控制 |
| LLM 限流 | 6 次/分钟/IP | 防止 LLM API 费用失控 |
| 精确缓存 | MD5 hash → 相同问题直接返回 | 不加语义判断，100% 精确防误杀 |
| 熔断器 | 连续 3 次失败 → 30s 冷却 | 防止雪崩 |

### 📊 监控管理

- **运行状态 API**（免认证）：`GET /api/status` → 实时请求量/缓存命中率/LLM 调用/费用估算
- **历史统计 API**：`GET /api/stats/history` → 每日统计数据
- **管理看板**（Basic Auth）：5 张 Chart.js 趋势图（请求量/费用/缓存命中率/LLM vs 缓存/反馈统计）+ 每日明细表格 + 高频未答问题建议区
- **定时持久化**：每 5 分钟自动保存统计快照

### 🔄 反馈飞轮

```
用户问题 → 帕鲁回答 → 用户赞/踩（Web 界面） → 记录到 feedback_log.json
                                                     ↓
用户问题 → 帕鲁答不上来 → 记录到 unanswered_log.json → 看板展示高频未答建议 → 手动补充知识库
```

- **赞/踩按钮**：Web 界面每条回答下方有 👍👎
- **反馈统计**：看板展示每日赞/踩趋势图
- **知识库建议**：`GET /api/unanswered/suggestions` 列出高频未答问题，提示补充知识库

### 🔌 API 端点

| 路径 | 方法 | 说明 | 认证 |
|:-----|:-----|:------|:-----|
| `/` | GET | Web 聊天界面 | 免认证 |
| `/admin` | GET | 管理看板（Chart.js 图表 + 明细） | Basic Auth |
| `/api/ask` | POST | 核心问答接口 | X-API-Key |
| `/api/status` | GET | 实时运行状态 | 免认证 |
| `/api/stats/history` | GET | 每日统计数据 | 免认证 |
| `/api/feedback` | GET/POST | 用户反馈（赞/踩） | 免认证 |
| `/api/unanswered/suggestions` | GET | 高频未答建议 | 免认证 |
| `/api/reload` | POST | 手动热重载 | X-API-Key |
| `/dingtalk/callback` | POST | 钉钉机器人回调 | 免认证 |

## 五、快速启动

### 前置条件

- Python 3.9+
- DeepSeek API Key（或兼容 OpenAI 接口的 LLM）
- （可选）Qwen-turbo API Key 作为降级模型

### 安装

```powershell
# 安装依赖
pip install openai flask waitress python-dotenv numpy

# 配 API Key（在项目目录下创建 .env）
DEEPSEEK_API_KEY=sk-your-key-here
```

### 启动

```powershell
# 方式一：前台调试
python app.py

# 方式二：后台运行（热更新守护进程）
.\start_palu.ps1 start     # 启动
.\start_palu.ps1 status    # 查看状态
.\start_palu.ps1 stop      # 停止
.\start_palu.ps1 reload    # 热重载
```

### 测试

```powershell
curl -X POST http://localhost:5000/api/ask `
  -H "Content-Type: application/json" `
  -H "X-API-Key: paludev2024" `
  -d '{"question": "软件闪退怎么办", "session_id": "test001"}'
```

## 九、项目总结

### 做了什么

5 天时间，从零做了一个生产级的 AI 售后客服机器人：

1. **Day 1**：调通 API 让帕鲁能说话（`palu.py`）
2. **Day 2**：灌了知识库让它能查文档回答（`palu_rag.py` + 36 条 FAQ）
3. **Day 3**：加了 ReAct Agent 让它会自己选工具查工单（`palu_agent.py`）
4. **Day 4**：接入钉钉群聊让 @机器人 能自动回复（`dingtalk_bot.py`）
5. **Day 5**：全套生产化加固——限流、缓存、安全防护、管理看板、API 认证、反馈飞轮（`app.py`）

### 关键数据

- 4000+ 请求处理，总花费 ~¥78
- 缓存命中率 ~65%（省了 65% 的 LLM 调用费）
- 单次问答成本 ~¥0.0009（含检索 + 生成）
- 知识库 36 条 FAQ，覆盖 8 个类别

### 核心原则

```
能检索的不生成 → 有缓存的不调用 → 能规则判断的不上 Agent → 无关问题不花钱
```
